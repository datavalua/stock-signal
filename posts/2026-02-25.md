늘어나는 데이터 수집량을 효율적으로 관리하기 위해 **듀얼 크롤러** 체제와 **배치(Batch)** 처리 방식을 전격 도입한 6일차 기록입니다. 또한, PC 환경에 구애받지 않고 일관된 개발 경험을 유지하기 위해 **Docker** 기반의 **Dev Container**를 구축하여 프로젝트 인프라를 한 단계 고도화했습니다. 최신 **google-genai** 라이브러리 연동으로 미래의 확장성까지 고려한 최적화 여정을 공유합니다.

먼저, 기존의 단일 크롤링 구조를 국내와 해외로 분리한 '듀얼 크롤러' 체제로 전환했습니다. 한국 거래소(KRX) 데이터는 장중 변동성을 반영하기 위해 정해진 간격으로 수집하고, 미국 시장 데이터는 나스닥 마감 시각에 맞춰 별도의 스케줄로 작동하도록 GitHub Actions 워크플로우를 최적화했습니다. 

이 과정에서 시도한 가장 큰 실험은 '배치(Batch) 처리 방식'의 도입입니다. 이전에는 종목마다 개별적으로 뉴스 요약을 요청했으나, API 호출 횟수를 줄여 비용을 절감하기 위해 유사한 성격의 리스트를 묶어 한 번에 처리하도록 로직을 개선했습니다. 초기 테스트에서는 호출 횟수를 80% 가량 줄이며 속도를 5배 이상 높이는 성과를 거두었으나, 이후 데이터 품질을 정밀하게 검토하면서 이 방식이 가진 한계를 마주하게 됩니다. (이후 더 높은 퀄리티를 위해 개별 요약 방식으로 다시 회귀하게 되는 중요한 기술적 복선이 됩니다.)

### 환경 표준화: Dev Container와 Docker 구축

개발 환경 측면에서는 'Dev Container(Docker)' 기술을 도입하여 환경 구축의 번거로움을 해결했습니다. 여러 대의 PC에서 작업하다 보면 파이썬 버전이나 패키지 의존성 문제로 인해 "내 컴퓨터에서는 되는데 다른 환경에서는 안 되는" 상황을 자주 마주하게 됩니다. `.devcontainer/devcontainer.json` 설정 파일을 통해 VS Code 환경에서 클릭 한 번으로 도커 컨테이너 기반의 표준화된 개발 환경이 구성되도록 구현했습니다. 이로써 팀 협업이나 환경 이전 시 발생할 수 있는 소모적인 디버깅 시간을 획기적으로 줄일 수 있게 되었습니다. 

### 확장성 확보: google-genai 비동기 처리 구조

또한, Gemini API 연동 라이브러리의 최신 버전인 `google-genai`를 연동하는 과정에서 비동기 처리 구조를 선제적으로 도입했습니다. 당장 대규모 트래픽이 발생하는 것은 아니지만, 데이터 수집량이 점진적으로 늘어날 미래를 대비하여 코드의 확장성과 안정성을 동시에 확보하는 데 주력했습니다.

이제 시스템은 단순히 작동하는 수준을 넘어, 대용량 데이터를 안정적으로 처리하고 어떤 환경에서도 재현 가능한 수준의 기술적 완성도를 갖추게 되었습니다. 자동화 파이프라인이 견고해짐에 따라 데이터 품질에 더 집중할 수 있는 여력이 생겼습니다. 내일은 축적된 주식 메타데이터에서 발견된 중복과 충돌 사례를 해결하고, 데이터 무결성을 확보하기 위한 정제 과정을 상세히 공유하겠습니다.
