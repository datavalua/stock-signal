"진정한 시스템의 완성도는 멈춰야 할 때를 아는 것에서 결정된다."

**토스 증권 시그널 클론 코딩** 프로젝트의 11일 차. 우리는 그동안 '어떻게 빠르고 많이 모을 것인가'에 집중했습니다. 하지만 오늘은 '어떻게 멈추고 올바르게 우회할 것인가'에 대한 이야기를 해보려 합니다. 주식 시장은 1년 365일 열리지 않습니다. 주말은 물론이고 국가 공휴일에 따라 변칙적으로 쉬어갑니다. 완벽해 보였던 자동 크롤링 시스템이 3월 초 연휴를 맞이하며 뜻밖의 오류를 뱉어낸 것입니다. 오늘의 기록은 이 '휴장일'이라는 예외 상황을 완벽하게 통제하고, 오히려 사용자 경험(UX)을 끌어올리는 기회로 삼은 트러블슈팅 과정입니다.

## 1 데이터 파이프라인의맹점:"장이 안 열리는 날의 크롤링"

GitHub Actions를 통해 매일 정해진 시간마다 크롤러가 작동하도록 스케줄링해두었습니다. 그리고 로컬 테스트 환경에서도 크롤러 로직 고도화가 이어졌습니다. 하지만 몇 가지 복병을 만났습니다.

### 1-1. 미국 주식 크롤링 오류 해결 (TypeError & Rate Limit)

한국 주식 데이터는 원활하게 수집되었지만, 미국 주식(US) 증시 크롤링 과정에서 두 가지 문제가 발견되었습니다.

- **인자 순서 오류 (TypeError)**: 미국 뉴스 중 가장 영향력을 가진 기사를 고르는 과정에서 `select_impactful_article` 함수를 호출할 때 인자 위치를 잘못 선언하여 `TypeError`가 발생했습니다. 해당 변수의 순서를 `(name, articles, change_val)`로 재배치하여 AI가 올바르게 작동하도록 고쳤습니다.
- **429 Quota Exceeded (요청량 초과)**: 미국 증시는 한국보다도 더 잦은 데이터를 끌어오다 보니 Gemini API Rate Limit (RPM 제한)에 걸리기 일쑤였습니다. API 호출마다 사이에 `time.sleep(15)`(15초 대기) 방어 로직을 삽입하여 RPM 제한을 극복했습니다.

### 1-2. 무의미한 펄스 시그널

주말이나 공휴일처럼 시장이 열리지 않는 날에도 크롤러는 성실하게 작동했습니다. 이로 인해 전야 종가 대비 변동(Change Rate)이 0%인데도 과거 찌꺼기를 가져와 '핫한 종목'으로 펄스(False) 시그널을 만들었습니다. 이는 데이터 신뢰도를 치명적으로 떨어뜨립니다. 삼일절과 같은 평일 공휴일과 각국 시장의 복잡한 휴장일 대처 방안이 필요했습니다.

## 2. 'holidays' 라이브러리 도입과 국가별 캘린더 연동

이 문제를 근본적으로 해결하기 위해 파이썬 생태계의 강력한 무기인 `holidays` 라이브러리를 도입했습니다. 이 라이브러리는 전 세계 주요 국가의 공식 휴일 데이터를 연도별로 정확하게 제공합니다.

```python
import holidays
from datetime import timedelta

# 한국과 미국 휴일 캘린더 로드
kr_holidays = holidays.KR()
us_holidays = holidays.US()

def is_holiday(check_date, market="KR"):
    # 주말 우선 필터링
    if check_date.weekday() > 4: 
        return True
    
    date_str = check_date.strftime("%Y-%m-%d")
    
    # 국가별 공휴일 필터링 (한국은 12월 31일 폐장일 로직 추가)
    if market == "KR":
        if check_date.month == 12 and check_date.day == 31: 
            return True
        return date_str in kr_holidays
    else:
        return date_str in us_holidays
```

이 로직을 크롤러의 핵심 파이프라인 첫 관문에 배치했습니다. 이제 크롤러는 작동을 시작하기 전에 "오늘이 해당 국가의 장이 열리는 날인가?"를 먼저 묻습니다. 만약 휴장일이라면, 즉시 무의미한 API 호출과 데이터 수집을 중단하고 프로세스를 종료합니다. 시스템이 스스로 '쉬어야 할 때'를 지능적으로 판단하게 된 것입니다.

## 3. 백워드 서치(Backward Search): 최신 영업일로의 타임머신

하지만 단순히 수집을 멈추는 것만으로는 부족했습니다. 클라이언트(프론트엔드)에서는 오늘 날짜의 데이터를 요청할 텐데, 서버에 데이터가 없으면 빈 화면이나 에러를 뿜어낼 테니까요. 그래서 도입한 것이 **'최신 영업일 백워드 서치(Backward Search)'** 로직입니다.

만약 오늘(`target_date`)이 휴일이라면, 시스템은 과거로 타임머신을 탑니다. 하루씩 뒤로 돌아가며(`check_date -= timedelta(days=1)`) 장이 열렸던 '가장 최근 영업일'을 찾을 때까지 탐색을 계속합니다. 

프론트엔드는 이 로직이 찾아낸 '진짜 마지막 거래일'의 데이터셋(`2026-02-27.json` 등)을 끌어오게 됩니다.

## 4. 예외를 기회로: 휴장일 알림 UX/UI 디자인

백엔드의 정합성을 맞춘 후, 사용자 경험(UX) 관점에서 빈틈을 채웠습니다. 사용자가 휴장일에 앱을 열었을 때 아무 설명 없이 어제 날짜 데이터만 보인다면 '오류인가?' 하고 당황할 수 있습니다. 이를 명확히 인지시키기 위해 Streamlit 앱 상단에 직관적인 알림 배너를 디자인하여 적용했습니다. (본문의 첨부 이미지를 참조해 주세요.)

![휴장일 안내 UI 디자인](/posts/images/2026-03-02.png)

적용된 코드는 다음과 같습니다.
`st.info("💡 오늘은 주식 시장 휴장일(주말/공휴일)입니다. 직전 개장일의 데이터를 표시합니다.", icon="⏸️")`

이 작은 디테일 하나가 서비스의 신뢰도를 수직 상승시켰습니다. 어설픈 에러 화면이나 침묵 대신, 상황을 투명하게 안내하고 가장 유용한 대체 정보(직전 거래일)를 우아하게 제공하는 것. 이것이 바로 기계적인 '수집기'와 체온이 담긴 '서비스'의 차이임을 깨달았습니다.

## 기록을 마치며

크롤링을 어떻게 더 정교하게 할 것인가만 치열하게 고민하다가, "언제 크롤링을 멈출 것인가"라는 전혀 다른 차원의 문제에 부딪히며 시야가 크게 확장된 하루였습니다. 시스템의 완성은 더하는 것보다 올바르게 빼는 것에서 이루어진다는 사실을 휴장일 트러블슈팅을 통해 배웠습니다. 어쩌면 우리 개발자들에게도, 무작정 달리는 것보다 휴일의 예외처리 로직처럼 스스로를 위한 '적절한 멈춤'이 필요한 것은 아닐까요? 이로써 파이썬 기반 백엔드의 단단한 뼈대 공사가 완벽히 마무리되었습니다.
