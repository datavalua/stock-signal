**토스 증권 시그널 클론 코딩** 프로젝트가 드디어 10일 차라는 중대한 이정표에 도달했습니다. 어제 구현한 '모델 이원화 전략'을 기반으로, 오늘은 시스템의 구조적 안정성을 확보하고 AI 요약의 품질을 극대화하는 '디테일의 완성'형 작업을 진행했습니다. 10일간의 여정을 돌아보며 마주한 예기치 못한 트러블슈팅과 성능 최적화의 기록입니다.

## 1. 구조 정리와 뜻밖의 배포 오류 (Troubleshooting)

프로젝트가 커짐에 따라 코드의 유지보수성을 높이기 위해 디렉터리 구조를 `backend/`와 `data/` 등으로 체계적으로 정리했습니다. 그런데 여기서 예상치 못한 복병을 만났습니다.

### 문제: "The main module file does not exist"
구조를 정리하고 나니 멀쩡히 돌아가던 Streamlit Cloud에서 배포 오류가 발생했습니다. Streamlit Cloud는 기본적으로 저장소 루트에 있는 `app.py`를 실행 파일로 기대하는데, 내부 구조를 바꾸면서 경로를 찾지 못하게 된 것입니다.

### 해결: 유연한 경로 로직 도입
`app.py`를 다시 루트로 이동시키되, 내부에서 `backend` 모듈을 불러올 때 경로 이슈가 발생하지 않도록 로직을 보강했습니다.

```python
# app.py 상단의 유연한 경로 로직
import os, sys
sys.path.append(os.path.dirname(__file__))
from backend import crawler
```

기존의 경직된 상대 경로 대신 파일 위치를 기반으로 한 동적 경로 추가 방식을 사용하여, 로컬과 클라우드 배포 환경 어디에서나 모듈 참조 오류 없이 안정적으로 작동하도록 만들었습니다.

## 2. 미국 주식 크롤링 오류 해결 (TypeError & Rate Limit)

한국 주식 데이터는 원활하게 수집되었지만, 미국 주식(US) 크롤링 과정에서 두 가지 치명적인 문제가 발견되었습니다.

### 문제 1: 인자 순서 오류 (TypeError)
미국 뉴스 기사 중 가장 영향력 있는 기사를 고르는 `select_impactful_article` 함수를 호출할 때 인자의 순서가 잘못 전달되어 `TypeError` 로그가 쏟아졌습니다. 객체가 아닌 문자열 배열이 위치하면서 기능이 멈춰버린 것입니다. 이를 `(name, articles, change_val)` 순서로 올바르게 재배치하여 AI가 기사 전문을 정확하게 인식하도록 수정했습니다. 

### 문제 2: 429 Quota Exceeded (요청량 초과)
또한 로컬 환경에서 미국 뉴스를 스크래핑할 때 짧은 시간 내에 다량의 API 요청이 발생하여 Gemini의 Rate Limit(RPM 제한)에 걸리는 문제가 지속되었습니다. 이를 해결하기 위해 API 호출 사이에 `time.sleep(15)`(15초 대기) 방어 로직을 삽입하여, 요청 속도를 의도적으로 조절하며 안정적으로 전체 사이클을 마칠 수 있도록 개선했습니다.

## 3. 할당량(Quota) 방어 작전: Tiered Smart Skip

GitHub Actions를 통해 20분 간격으로 크롤링을 수행하다 보니, 새로운 뉴스가 없음에도 중복해서 API를 호출하는 쿼터 낭비 문제가 대두되었습니다. 구글 Gemini API의 무료 티어(Free Tier)를 사용하고 있기 때문에, 위에서 겪은 RPM 제한 외에도 일일 한도(RPD) 관리가 필수적이었습니다. 이를 해결하기 위해 **'계층형 스킵 로직(Tiered Skip Logic)'**을 고안하여 전면 적용했습니다.

- **Tier 1 (Instant Skip - Heuristic)**: 기사 목록을 스크래핑한 직후, 가장 최신 기사의 헤드라인(Title)과 URL을 이전 크롤링 사이클에서 저장했던 JSON 데이터의 최신 기사와 1:1로 비교합니다. 만약 완전히 동일하다면, 이는 해당 주식에 대해 새로운 소식이 없다는 명백한 증거입니다. 이 경우 비용이 발생하는 AI 요약 API를 아예 호출하지 않고, 기존에 작성해둔 요약문을 그대로 재사용(캐싱)하여 다음 종목으로 넘어갑니다.
- **Tier 2 (AI-Assisted Skip)**: Tier 1 방어선을 통과했더라도(새로운 자잘한 기사가 떴더라도), 1차적으로 AI가 수집된 전체 기사 목록 중 *가장 주가 변동에 큰 영향을 미쳤을 법한 핵심 기사 1개를 선정*하는 과정을 거칩니다. 만약 여기서 AI가 '최적의 기사'로 뽑아든 것이 알고 보니 이전 사이클에서 이미 요약에 사용했던 기사라면 어떻게 될까요? 새로운 기사가 뜨긴 했어도 주가에 미치는 핵심적인 이슈는 변함이 없다는 뜻입니다. 이 경우에도 막대한 토큰이 소모되는 '전체 기사 종합 요약' API 호출을 과감히 생략하고 기존 데이터를 유지합니다.

이러한 지능형 2중 스킵 시스템을 한국 및 미국 증시 모두에 완벽하게 호환되도록 정리한 덕분에, 아무리 자주 크롤링 루틴이 돌아도 실제 API 호출은 '주가에 영향을 줄 만한 진짜 새로운 정보'가 떴을 때만 발생하게 되었습니다.

결과적으로 할당량 초과 에러 없이 안정적으로 퀄리티 높은 요약본을 자동 생성하는 비용 효율적인 파이프라인을 완성해냈습니다. 효율적인 아키텍처 설계가 인프라 비용을 어떻게 절감할 수 있는지 여실히 보여주는 사례였습니다.
