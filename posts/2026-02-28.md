서비스 운영의 지속 가능성을 확보하기 위해 **Gemini API**의 할당량(Quota) 문제를 해결한 9일차 여정입니다. 잦은 호출로 인한 **429 Error**를 방어하기 위해 하루 1,500회의 넉넉한 한도를 제공하는 **`gemini-flash-lite-latest`** 모델을 전사적으로 도입했습니다. 이를 통해 로컬 개발 환경과 **GitHub Actions** 자동화 서버 모두에서 중단 없는 데이터 수집 체계를 완성했습니다.

### 운영 비용의 압박: API 할당량(Quota) 문제

무료 티어 기반의 Gemini API는 분당 호출 횟수(RPM)와 일일 호출 한도에 명확한 제약이 존재합니다. 개발 초기에는 호출량이 많지 않아 문제가 되지 않았으나, 뉴스 검색 결과에 따른 배치 요약 요청이 누적되면서 정작 실제 배포 환경에서 할당량 부족으로 시그널이 생성되지 않는 현상이 발생하기 시작했습니다. 로컬 디버깅을 위한 무분별한 API 호출이 운영 서버의 가용성을 갉아먹고 있었던 것입니다.

### 해결책: 전면적인 `gemini-flash-lite-latest` 모델 전사 도입

이러한 문제를 구조적으로 해결하기 위해 '운영/로컬 환경별 전용 모델 이원화 전략'을 한때 고민하기도 했습니다. 그러나 관리 포인트만 늘어나고 근본적인 쿼터 부족을 해소하기는 어려웠습니다. 결국, 실행 환경에 구애받지 않고 **가장 넉넉한 할당량을 제공하는 `gemini-flash-lite-latest` 모델을 크롤러 전체에 전면적으로 도입**하는 전략을 채택했습니다.

### 구현: 명시적인 고한도 모델 고정 로직

처음에는 단순히 `gemini-2.5-flash`를 쓰면 될 줄 알았습니다. 하지만 구글의 무료 정책이 변경되면서 해당 모델의 하루 최대 한도가 20~50회에 불과해 계속해서 429 Error(한도 초과)가 발생하는 아찔한 경험을 했습니다. 추가로 `gemini-2.0-flash` 모델 역시 무료 티어에서는 토큰 허용량이 0으로 제한되어 있는 황당한 상황도 마주했죠.

수많은 시행착오 끝에, 저희가 찾은 최적의 크롤링 모델은 다음과 같습니다:

```python
# 환경에 얽매이지 않고 고정적인 고한도 모델을 선택하는 로직 (crawler.py)
import os

def get_model_name():
    """Select model based on environment (GitHub Actions vs Local)."""
    # 사용자가 환경 변수(GEMINI_MODEL)로 명시적으로 지정한 모델이 있다면 최우선 적용합니다.
    env_model = os.getenv("GEMINI_MODEL")
    if env_model:
        return env_model
        
    # User requested to always use the high-quota model for daily stock signal JSON generation
    # `gemini-flash-lite-latest` has a limit of 1,500 RPD (Requests Per Day) on the free tier.
    return 'gemini-flash-lite-latest'
```

### 결과: 환경 제약을 넘어선 안정적인 서비스 운영

이러한 고정적인 `gemini-flash-lite-latest` 전략은 극적인 긍정적 효과를 가져왔습니다.

운영 안정성의 절대적 확보입니다. 로컬과 GitHub Actions 자동화 배포 환경 모두에서 하루 1,500회라는 넉넉한 호출 한도를 확보했습니다. 한국 주식 10개, 미국 주식 10개를 매일 몇 번이나 구동해도 하루 약 40~100회의 API만 호출하므로, 전체 할당량의 5% 내외만 사용하여 쿼터 부족 오류 없이 매우 안정적으로 시그널을 생성할 수 있게 되었습니다.

또한 모델 전환과정에서 발생할 수 있는 응답 객체의 파싱 에러를 방어하고, JSON 포맷을 안정적으로 출력하도록 프롬프트 구조까지 개선하며 데이터 일관성을 유지했습니다. 

효율적인 자원 관리는 기술적 완성도를 넘어 프로젝트의 지속 가능성을 결정짓는 중요한 요소입니다. 무조건 단일 최고 성능의 도구만 고집하기보다 프로젝트의 가장 중요한 목적인 '끊임없는 데이터 아카이빙'에 맞춰 가장 할당량이 많은 도구를 선택하는 것이 실무에서 얼마나 큰 가치를 발휘하는지 다시금 깊이 체감했습니다.
