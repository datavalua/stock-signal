프로젝트의 기능 구현이 막바지에 이르면서, 실제 운영 중인 시스템의 안정성과 개발 효율성을 동시에 확보해야 하는 과제에 직면했습니다. 특히 Gemini API와 같은 클라우드 기반 AI 서비스를 활용할 때 발생할 수 있는 할당량(Quota) 제한 문제는 개발 생산성에 직접적인 영향을 미칩니다. 오늘은 운영 환경의 안정성을 해치지 않으면서 로컬 개발 환경에서 효율적으로 모델을 테스트하고 최적화하는 전략을 구현했습니다.

## 운영 비용의 압박: API 할당량(Quota) 문제

무료 티어 기반의 Gemini API는 분당 호출 횟수(RPM)와 일일 호출 한도에 명확한 제약이 존재합니다. 개발 초기에는 호출량이 많지 않아 문제가 되지 않았으나, 뉴스 검색 결과에 따른 배치 요약 요청이 누적되면서 정작 실제 배포 환경에서 할당량 부족으로 시그널이 생성되지 않는 현상이 발생하기 시작했습니다. 로컬 디버깅을 위한 무분별한 API 호출이 운영 서버의 가용성을 갉아먹고 있었던 것입니다.

## 해결책: 환경별 모델 이원화 전략 도입

이러한 문제를 구조적으로 해결하기 위해 '환경별 전용 모델 이원화 전략'을 도입했습니다. 핵심 아이디어는 실행 환경이 로컬 호스트인지 아니면 GitHub Actions 서버인지에 따라 서로 다른 성능과 비용 정책을 가진 모델을 동적으로 배정하는 것입니다. 운영 환경에서는 최고의 추출 성능과 한국어 이해도를 보장해야 하므로 `gemini-2.5-pro` 모델을 고정적으로 사용하고, 쿼터 보존이 중요한 로컬 개발 환경에서는 응답 속도가 빠르고 할당량이 상대적으로 넉넉한 `gemini-2.5-flash` 모델을 기본값으로 설정했습니다.

## 구현: 환경 감지 및 모델 선택 로직

```python
# 환경에 따른 Gemini 모델 선택 로직 (crawler.py)
import os

def get_model_name():
    """
    실행 환경(GitHub Actions vs 로컬)에 따라 최적의 모델을 선택합니다.
    """
    # 1. 사용자가 명시적으로 지정한 모델이 있다면 최우선 적용
    env_model = os.getenv("GEMINI_MODEL")
    if env_model:
        return env_model
        
    # 2. GITHUB_ACTIONS는 GitHub Actions 환경의 예약된 환경 변수입니다.
    # 운영(CI) 환경에서는 할당량이 넉넉한 Flash 모델을 사용합니다.
    if os.getenv('GITHUB_ACTIONS') == 'true':
        return 'gemini-flash-latest' # 1,500 RPD
        
    # 3. 로컬 환경에서는 품질을 위해 Pro 모델을 기본으로 사용합니다.
    return 'gemini-pro-latest' # 50 RPD
```

## 결과: 안정적인 서비스 운영과 품질 확보

이러한 모델 이원화 전략은 두 가지 측면에서 긍정적인 효과를 가져왔습니다. 첫째, 운영 안정성의 확보입니다. GitHub Actions를 통한 자동화 배포 환경에서는 `gemini-flash-latest`를 사용하여 하루 1,500회라는 넉넉한 호출 한도를 확보했습니다. 덕분에 여러 번의 크롤링이 돌아가도 쿼터 부족 오류 없이 안정적으로 시그널을 생성할 수 있게 되었습니다.

둘째, 로컬 개발 시의 품질 최적화입니다. 로컬에서는 호출 횟수가 많지 않으므로, 호출 한도는 적지만 성능이 더 뛰어난 `gemini-pro-latest` 모델을 사용하여 요약의 결과물과 로직을 더 정교하게 다듬을 수 있습니다. 

또한 모델 전환 과정에서 발생할 수 있는 응답 객체의 스키마 차이를 방어하기 위해, 요약 결과물을 파싱하는 로직에 정규화(Normalization) 과정을 추가했습니다. 모델의 종류에 관계없이 시스템은 동일한 JSON 포맷을 출력하도록 설계하여 데이터 일관성을 유지했습니다. 

효율적인 자원 관리는 기술적 완성도를 넘어 프로젝트의 지속 가능성을 결정짓는 중요한 요소입니다. 무조건 최고 성능의 도구만 고집하기보다 상황에 맞는 최적의 도구를 조합해 사용하는 유연함이 실무에서 얼마나 큰 가치를 발휘하는지 다시금 체감했습니다. 내일은 프로젝트의 안정화 단계를 거쳐, 데이터 아카이브를 넘어선 더 큰 도약(Next.js 등으로의 확장)을 준비하는 과정을 다루겠습니다.
