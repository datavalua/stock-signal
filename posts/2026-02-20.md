## 자동 실행 환경 구축: GitHub Actions 활용

가장 먼저 고려한 요소는 '자동 실행 환경'입니다. 매일 특정 시간에 스크립트를 수동으로 작동시키는 것은 효율적이지 않으므로, GitHub Actions의 cron 스케줄 기능을 도입하기로 했습니다. GitHub Actions는 코드 빌드나 테스트 용도로 널리 사용되지만, 주기적인 작업 실행과 결과물의 자동 커밋 기능을 활용하면 서버 없이도 훌륭한 데이터 엔진 역할을 수행할 수 있습니다. 

## 데이터 수집 라이브러리: FinanceDataReader 선정

데이터 수집 라이브러리는 신뢰도가 높은 FinanceDataReader를 선정했습니다. FinanceDataReader는 한국거래소(KRX)뿐만 아니라 해외 주식 데이터까지 통합적으로 제공하여 추후 확장성 면에서 유리합니다. 개발 과정에서 KRX 전체 종목을 리스팅하는 `fdr.StockListing("KRX")` 호출 시 기대와 다른 컬럼 데이터가 반환되는 상황을 발견했습니다. 이는 라이브러리 내부의 데이터 파싱 방식이 변경되었을 가능성이 높으므로, 즉시 별도의 테스트 파일을 작성하여 데이터 스키마를 확인했습니다.

```python
# FinanceDataReader 종목 리스팅 검증 코드
import FinanceDataReader as fdr
import traceback

print("Testing fdr.StockListing('KRX')...")
try:
    df_krx = fdr.StockListing("KRX")
    print("Listing success. Shape:", df_krx.shape)
    print("Columns in data:", df_krx.columns.tolist())
    # 시장 구분(KOSPI/KOSDAQ) 정보 포함 여부 확인
    if 'Market' in df_krx.columns:
        print("Market types found:", df_krx['Market'].unique().tolist())
    else:
        print("Market column missing in the current version.")
except Exception as e:
    print("Listing access failed:", str(e))
    traceback.print_exc()
```

위의 점검을 통해 시장 구분 값과 종목 코드, 명칭을 정확히 식별하여 수집할 수 있는 로직을 구성했습니다. 직접 데이터를 출력하며 명세를 확인하는 과정이 공식 문서를 살피는 것보다 명확한 해결책을 제시할 때가 많음을 다시 확인했습니다.

## AI 페어 프로그래밍: Anti Gravity와의 협업

또한 개발 단계에서 가장 큰 조력자는 AI 코딩 어시스턴트인 'Anti Gravity'였습니다. 프로젝트의 초기 뼈대를 잡고, 특히 파이썬 스크립트를 GitHub Actions라는 제한적인 환경에서 어떻게 효율적으로 구동할지에 대한 정교한 가이드를 제공받았습니다. 복잡한 환경 변수 처리와 종목 리스팅 결과에 따른 동적 루프 구성 등, 기획 단계에서 놓치기 쉬운 기술적 디테일들을 Anti Gravity와의 대화를 통해 꼼꼼히 채워 넣을 수 있었습니다.

## 분석 엔진 기초 설계: Gemini 1.5 Pro API

모델 연동 및 AI 분석 엔진의 기초 설계에는 'Gemini 1.5 Pro' API를 사용하기로 결정했습니다. 주식 뉴스는 복합적인 경제 맥락과 기업 고유의 기술 용어가 혼재되어 있어, 이를 정확히 이해하고 압축하기 위해서는 고성능 대형 언어 모델의 도움이 필수적이었습니다. 초기 단계에서는 모델의 효율성보다는 뉴스 본문의 뉘앙스를 얼마나 정확히 포착하는지에 초점을 맞추어 텍스트 분석 로직을 검증했습니다. 

## 데이터 무결성: 종목 코드 제로 패딩(Zero-padding) 이슈

초기 수집 로직 구현 중 가장 끈질기게 괴롭혔던 문제는 주식 종목 코드의 제로 패딩(Zero-padding) 처리였습니다. '005930'과 같은 코드가 파이썬 데이터프레임으로 로드되는 과정에서 숫자로 오인되어 '5930'으로 잘려 나가는 현상이 발생했습니다. 이를 방지하기 위해 `dtype`을 명시적으로 문자열로 지정하고, 모든 수집 파이프라인에서 6자리 코드를 엄격히 보존하도록 검증 로직을 추가했습니다. 안정적인 데이터 처리를 위한 가장 기초적인 단계였지만, 이 작은 디테일이 데이터 아카이브의 신뢰도를 결정짓는 핵심임을 깨달았습니다.

Gemini API 연동 시에는 신규 출시된 `google-genai` 라이브러리를 채택했습니다. 모델과의 안정적인 통신을 위해 `ImportError`와 네트워크 지연 등 예외 상황에 대한 처리를 강화했습니다.

프로젝트의 엔진 역할을 하게 될 Gemini API 연동 결과는 다음과 같이 JSON 형태로 확인했습니다. 별도의 화려한 UI 없이, 로직의 정확성에만 집중하던 시기였습니다.

```json
{
  "symbol": "005930",
  "name": "삼성전자",
  "date": "2026-02-20",
  "signal": {
    "theme": "반도체/HBM",
    "summary": "엔비디아 서플라이 체인 편입 기대감에 따른 외국인 매수세 유입",
    "impact_score": 4.5
  }
}
```

위 텍스트는 성공적으로 수행된 첫 번째 API 테스트 결과입니다. 비록 짧은 응답이었으나, 시스템의 두뇌 역할을 할 AI 연동이 정상적으로 작동함을 확인한 지점입니다. 이제 기초적인 수집과 실행 환경은 갖추어졌습니다. 내일부터는 수집된 수천 개의 뉴스 기사 속에서 Gemini가 어떻게 핵심 줄기를 찾아내게 할지, 요약 알고리즘을 정교화하는 과제가 기다리고 있습니다.
